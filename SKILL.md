# 편백나무 몽둥이 (Pbeonbat Mongdungi)

> *편백나무(Hinoki)* — 단단하면서 유연한 재목. 오래 쓸수록 빛나는 도구의 뼈대.
> *몽둥이* — 명확한 한 방. 흐릿한 LLM 습관을 한 방에 잡는 규율.

LLM은 코드를 짤 때 세 가지 병을 반복한다: (1) 묻지 않은 것까지 구현하고, (2) 인접한 코드를 "개선"하며, (3) 가정을 숨긴 채 앞으로 나아간다. 이 스킬은 그 병을 두 개의 축으로 잡는다.

- **몽둥이**: Andrej Karpathy의 CLAUDE.md에 담긴 4원칙. 스스로를 매 단계마다 점검하는 규율.
- **편백나무 장인**: 로컬에 설치된 Claude Code CLI와 Codex CLI. 검증·구현·리뷰를 위임하는 제2의 시니어 엔지니어.

둘 다 있어야 한다. 규율 없는 장인은 방향을 잃고, 장인 없는 규율은 힘이 빠진다.

---

## 언제 이 스킬이 개입하는가

다음 시점 중 하나라도 해당하면 이 스킬의 지시를 따른다.

- 코드를 한 줄이라도 쓰기 직전
- 새 함수/클래스/파일을 만들기 전
- 기존 코드를 편집하려 할 때
- "이거 맞게 했는지 확인해줘" / "리뷰해줘" 같은 자문 요청
- 사용자의 요청이 2줄 이하로 모호할 때
- 자신이 작성한 diff가 30줄을 넘을 것 같을 때

그 외에도, 작업 도중 "이게 최선인가?" 스스로 묻게 되는 순간에 이 스킬로 돌아와라.

## 핵심: 4원칙 자기 점검

이 원칙들은 Karpathy가 수천 개 PR을 관통하며 정리한 LLM 코딩 실수의 패턴을 역으로 적어놓은 것이다. **왜** 이 원칙이 필요한지 이해하면, 규칙이 아니라 본능이 된다.

### 원칙 1 — Think Before Coding (코딩 전에 생각하라)

**문제**: LLM은 "일단 짜고 보자"는 경향이 있다. 코드를 쓰기 시작하면 가정이 굳어져서, 틀린 가정 위로 200줄이 쌓인다.

**실행**: 코드를 쓰기 전에 반드시:

1. **가정을 입 밖로 낸다.** "나는 `user.id`가 항상 존재한다고 가정하고 있어." 이 한 줄이 문제의 절반을 해결한다. 가정이 틀렸다면 사용자가 고쳐주고, 맞다면 명확해진다.
2. **여러 해석이 있으면 조용히 고르지 않는다.** "A로 해석해서 X를 할 수도 있고, B로 해석해서 Y를 할 수도 있어. 어느 쪽이야?" 한 번의 클리어가 세 번의 재작업을 막는다.
3. **더 단순한 방법이 있으면 말한다.** 사용자가 "마이크로서비스로 만들어달라"고 해도, 단일 파일로 끝날 일이면 "사실 이건 하나의 스크립트로 충분합니다"라고 말한다. 시니어 엔지니어는 요청을 그대로 따르기보다 목적을 따른다.
4. **모르는 게 있으면 멈춘다.** "이 부분이 이해가 안 돼요. [구체적으로 뭘 모르겠는지]." 대충 짐작해서 채우지 않는다.

**왜**: 코딩 10분 전의 1분 질문이 코딩 1시간 후의 30분 디버깅을 줄인다. 생각은 싸고, 재작업은 비싸다.

### 원칙 2 — Simplicity First (단순성 우선)

**문제**: LLM은 "이건 나중에 확장할 수도 있으니까"라며 추상화·설정 가능성·플러그인 시스템을 끼워넣는다. 그 90%는 영원히 쓰이지 않지만, 코드를 읽기 어렵게 만든다.

**실행**: 문제를 푸는 **최소 코드**만 쓴다.

- 요청받지 않은 기능은 추가하지 않는다.
- 한 번만 쓰이는 코드에 추상화·인터페이스·팩토리를 끼우지 않는다.
- "유연성", "구성 가능성", "재사용성"을 요청받지 않았으면 만들지 않는다.
- 일어날 수 없는 시나리오의 에러 핸들링을 쓰지 않는다.
- 200줄을 썼는데 50줄로 줄일 수 있다면, 다시 쓴다.

**자문**: "이 코드를 시니어 엔지니어가 리뷰하면 '왜 이렇게 복잡해?'라고 할까?" — 그렇다면 줄인다.

**왜**: 한 번 쓰이고 말 코드에 구조를 만들면, 그 구조를 읽는 데 드는 시간이 코드를 쓴 시간보다 길어진다. 단순함은 미덕이 아니라 경제다.

### 원칙 3 — Surgical Changes (수술적 변경)

**문제**: LLM은 손댄 파일에서 "이것도 정리하면 좋겠다"며 주변을 휩쓸어버린다. diff가 부풀어서 진짜 의도된 변경을 리뷰할 수 없게 된다.

**실행**: 손대야 하는 곳만 손댄다. 내가 만든 엉망만 내가 치운다.

기존 코드를 편집할 때:

- 인접한 코드, 주석, 포맷을 "개선"하지 않는다.
- 고장나지 않은 건 리팩토링하지 않는다.
- 내 스타일이 더 낫다고 해도 기존 스타일을 따른다.
- 무관한 데드 코드를 발견하면 지우지 말고 언급만 한다.

내 변경 때문에 고립된 것이 생기면:

- 내 변경이 안 쓰게 만든 import/변수/함수는 지운다. (내 엉망)
- 원래 있던 데드 코드는 요청받기 전엔 건드리지 않는다. (남의 엉망)

**테스트**: 변경된 모든 라인이 사용자의 원래 요청으로 직접 추적될 수 있어야 한다.

**왜**: diff는 커뮤니케이션이다. 리뷰어가 "이 줄은 왜 바뀌었지?"라고 물어야 한다면, 이미 실패한 diff다. 수술적 변경은 존중이다 — 코드를 함께 쓰는 사람들에 대한.

### 원칙 4 — Goal-Driven Execution (목표 주도 실행)

**문제**: "대충 만들어줘"는 무한 루프를 낳는다. 언제 끝났는지 모르니까 영원히 만진다.

**실행**: 작업을 **검증 가능한 목표**로 바꾼다.

| 흐릿한 목표 | 검증 가능한 목표 |
|---|---|
| "validation 추가" | "잘못된 입력에 대한 테스트를 먼저 쓰고, 그것들이 통과하도록 만든다" |
| "이 버그 잡아줘" | "버그를 재현하는 테스트를 먼저 쓰고, 그것이 통과하도록 만든다" |
| "X 리팩토링" | "리팩토링 전후로 모든 기존 테스트가 통과하는지 확인한다" |

다단계 작업이라면, 각 단계에 검증 지점을 건다:

```
1. [단계] → 검증: [체크]
2. [단계] → 검증: [체크]
3. [단계] → 검증: [체크]
```

강한 성공 기준은 혼자서 루프를 돌 수 있게 한다. 약한 기준("그냥 되게 해줘")은 계속 clarification을 요구하게 만든다.

**왜**: 명확한 끝이 있어야 멈출 수 있다. "완료"의 정의가 없으면 작업은 팽창한다 — 학교 과제처럼.

## 편백나무 장인: 로컬 Claude/Codex CLI 부리기

원칙은 내면의 규율이라면, CLI는 외부의 검증자다. 로컬에 Claude Code CLI와 Codex CLI가 깔려 있으면 — **반드시 활용한다**. 혼자 모든 걸 하려 하지 말고, 두 번째 의견을 받아라. 시니어 엔지니어도 PR을 올리기 전 동료에게 한 번 더 보여준다.

### 왜 로컬 CLI인가

- **빠르다**: 네트워크 왕복 없이 로컬에서 실행.
- **독립적이다**: 다른 컨텍스트, 다른 추론으로 보기 때문에 내가 놓친 걸 잡아준다.
- **권한 제어 가능**: `--allowedTools`로 읽기 전용 모드로 안전하게.
- **결정이 공개된다**: CLI가 준 답을 사용자에게 보여주고, 따를지 말지 결정하게 한다.

### CLI 디텍션

설치 위치가 표준 PATH 밖일 수 있다. `scripts/detect_cli.sh`가 알아서 찾는다 — 직접 경로를 하드코딩하지 마라.

```bash
bash scripts/detect_cli.sh
# 출력 예:
# CLAUDE_CLI=/path/to/claude
# CLAUDE_VERSION=2.1.170
# CODEX_CLI=/Applications/Codex.app/Contents/Resources/codex
# CODEX_VERSION=0.140.0-alpha.2
```

이 스크립트를 호출해 경로를 동적으로 받아라. 사용자가 다른 머신에서 쓰거나 업데이트해도 깨지지 않는다.

### 언제 무엇을 부르는가

| 상황 | 무엇을 | 어떻게 |
|---|---|---|
| **구현 전**: 두 가지 접근이 있고 어느 쪽이 맞는지 모를 때 | Claude (reasoning 강함) | 두 접근을 설명하고 tradeoff를 물어본다 |
| **구현 중**: 특정 함수 하나만 빠르게 구현해달라고 할 때 | Codex (가볍고 빠름) | 단일 함수 시그니처와 제약을 준다 |
| **구현 후**: diff 리뷰, surgical change 원칙 위반 확인 | Claude (리뷰에 강함) | diff와 함께 "원칙 3 위반하는 줄 찾아줘" |
| **버그**: 재현이 안 되고 원인을 모를 때 | Claude | 관련 파일과 에러를 주고 가설을 물어본다 |
| **복잡한 리팩토링 설계**: 영향도 파악 | Claude | 현재 코드 + 리팩토링 의도를 주고, 영향받는 지점을 물어본다 |
| **빠른 second opinion**: "이거 맞아?" | Codex | 1-2문장으로 물어본다 |

**원칙**: 복잡한 추론/설계/리뷰 → Claude, 빠른 구현/단순 질문 → Codex. 헷갈리면 Claude가 기본.

### 호출 패턴 (non-interactive)

```bash
# Claude Code - 한 번에 출력 (-p / --print 모드)
# --tools "" 는 모든 도구 비활성화 (순수 추론, 가장 안전)
"$CLAUDE_CLI" -p --tools "" "질문이나 프롬프트"
# 읽기 도구만 허용 (코드 리뷰용)
"$CLAUDE_CLI" -p --allowedTools "Read,Grep,Glob" --add-dir "$(pwd)" "코드 리뷰해줘"

# Codex - 비대화형 실행
# --skip-git-repo-check 는 git 레포 밖에서도 실행, --sandbox read-only 로 파일 변경 차단
"$CODEX_CLI" exec --skip-git-repo-check --sandbox read-only "질문이나 프롬프트"
```

상세한 호출법·플래그·안전 가이드는 `references/cli_usage.md`를 읽어라. 거기에:

- 읽기 전용으로 안전하게 호출하는 법 (`--tools ""`, `--sandbox read-only`)
- 파일 컨텍스트를 넘기는 법 (`--add-dir`, `-C`, stdin)
- 출력을 파이프로 받아 처리하는 법
- 권한/비용/속도 트레이드오프
- 호출 후 사용자에게 어떻게 보고할지

를 다룬다.

### 안전 규칙 (절대 어기지 말 것)

1. **CLI 결과를 블라인드로 적용하지 않는다.** CLI가 "이렇게 바꿔라" 하면, 사용자에게 보여주고 확인받는다. CLI도 LLM이다 — 틀릴 수 있다.
2. **파일을 쓰게 두지 않는다.** 읽기 전용 플래그를 켜고, 파일 변경은 항상 내가 직접 한다 (원칙 3: surgical).
3. **CLI 실패를 무시하지 않는다.** CLI가 없으면 없다고 솔직히 말하고, fallback으로 내가 직접 한다.
4. **비용을 의식한다.** Claude/Codex 호출은 돈이다. 한 줄 질문에 30초 호출을 낭비하지 않는다 — 진짜 가치 있는 검증에만.

## 작업 워크플로우

실제 작업에 이 스킬을 어떻게 끼워넣는지, 단계별로.

### Phase 0 — 코딩 전 (Think 단계)

1. **요청 분해**: 사용자가 진짜 원하는 게 뭔지 1문장으로 요약한다.
2. **가정 나열**: 내가 세운 가정을 적는다 ("X는 항상 Y라고 가정").
3. **단순 경로 확인**: "이걸 풀기 위한 가장 단순한 방법은?"을 먼저 묻는다.
4. **모호하면 멈춤**: 원칙 1의 실행 규칙 적용. 질문이 필요하면 이 시점에 한다.
5. *(선택)* 두 접근이 갈리면 → Claude CLI에 tradeoff를 물어본다.

**산출물**: 구현 계획 (짧게), 검증 기준 (원칙 4).

### Phase 1 — 구현 (Simplicity + Surgical 단계)

1. **계획대로만 쓴다.** 코딩하면서 새 기능이 떠오르면 메모만 해두고, 요청받지 않은 건 넣지 않는다.
2. **인접 코드를 건드리고 싶어지면 멈춘다.** "이것도 고치면 좋을텐데…" → 묻지 않는다. 변경 사항에 메모만 남긴다.
3. **매 30줄마다 점검**: "이 줄이 원래 요청으로 추적되나?" (원칙 3 테스트).

### Phase 2 — 검증 (Goal-Driven 단계)

1. **성공 기준 체크**: Phase 0에서 정한 검증 기준을 실행한다.
2. **테스트/빌드 실행**: 가능하면 직접 돌린다.
3. *(권장)* **CLI second opinion**: 완성된 diff를 Claude에게 주고, 원칙 3(surgical) 위반과 원칙 2(simplicity) 위반을 찾아달라고 한다.

### Phase 3 — 보고

1. **변경 요약**: 무엇을 바꿨는지, 왜 바꿨는지 3-5줄.
2. **메모만 한 것들**: "이런 것도 고칠 수 있었는데 안 했어. 할까?" 형태로 제안.
3. *(사용한 경우)* **CLI 결과 요약**: "Claude CLI에게 물어봤더니 X를 지적했어. 반영했어/안 했어." — 사용자가 판단할 수 있게.

## 자주 하는 실수와 치료법

LLM이 반복하는 패턴. 이 표를 보고 스스로를 점검한다.

| 실수 | 증상 | 치료 |
|---|---|---|
| **과잉 추상화** | 한 번 쓸 함수를 인터페이스로 만든다 | "이거 재사용될 일 있어?" 없으면 인라인 |
| **인접 정리병** | 고장 안 난 코드를 "개선" | "내 요청과 직접 연결돼?" 아니면 손 뗀다 |
| **숨은 가정** | "아마 이럴 거야" 짐작하고 코딩 | 가정을 입 밖로 낸다 (원칙 1) |
| **무한 다듬기** | "거의 다 됐는데"를 반복 | 성공 기준을 미리 정한다 (원칙 4) |
| **에러 핸들링 과잉** | 절대 안 일어날 예외 처리 | "이 시나리오 실제로 일어나?" 아니면 뺀다 |
| **CLI 블라인드 신뢰** | Claude CLI가 하라는 대로 함 | 항상 사용자 확인 거치기 |

## 빠른 참조: 원칙 한 줄 요약

코딩 중 헷갈리면 이 네 문장으로 돌아간다.

1. **Think** — 가정을 말로 꺼내라. 모호하면 멈춰라.
2. **Simplicity** — 최소 코드. 안 달라고 한 건 만들지 마라.
3. **Surgical** — 손댄 곳만. 주변은 건드리지 마라.
4. **Goal-driven** — 검증 가능한 끝. 그 전엔 멈추지 마라.
